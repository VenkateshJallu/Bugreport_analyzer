# Android BugReport Analyzer

> **Generic, multi-build, command-line tool for deep analysis of Android `bugreport.txt` files.**  
> Produces dark-themed HTML reports and PNG graphs for battery drain, ANRs, crashes, LMK, thermal events, and wakelocks.  
> Supports user-defined SLA thresholds so analysis verdicts reflect device/test-specific expectations rather than generic defaults.

## Quick Start

```bash
# 1 — Single bugreport
python bugreport_analyzer.py bugreport.txt

# 2 — Two builds (individual + comparative)
python bugreport_analyzer.py rc1.txt st5.txt -l RC1 ST5

# 3 — Output to a specific folder
python bugreport_analyzer.py rc1.txt st5.txt -l RC1 ST5 -o C:\Reports\Sprint42

# 4 — With device-specific SLA thresholds
python bugreport_analyzer.py bugreport.txt -l TC501L ^  
       --expected-hours 20 --expected-drain 35 --crit-drain 60
```

## CLI Reference

```
python bugreport_analyzer.py <path1> [path2 ...] [OPTIONS]
```

### Positional Arguments

| Argument | Description |
|---|---|
| `path1 [path2 ...]` | One or more paths to `bugreport*.txt` files (full or relative paths) |

### General Options

| Flag | Short | Default | Description |
|---|---|---|---|
| `--labels L1 L2 …` | `-l` | `Build1, Build2 …` | Short labels for each bugreport (must match count of paths) |
| `--output DIR` | `-o` | Directory of first bugreport | Output directory for all generated files |
| `--title "TEXT"` | `-t` | Auto-generated from labels + date | Title shown in the HTML report header |
| `--no-graph` | — | (graphs generated by default) | Skip PNG generation — faster when matplotlib is unavailable |
| `--help` | `-h` | — | Show help message |

### Analysis Thresholds

All threshold flags are **optional**. When omitted, the defaults shown below are used.  
When provided, they override every colour-coded KPI, bar chart, anomaly detection, and RCA verdict throughout the full report pipeline.

| Flag | Type | Default | Description |
|---|---|---|---|
| `--expected-hours H` | float | — | Expected session run duration in hours (e.g. `20.0`). Sessions shorter than 85 % of this value are flagged **CRITICAL** in the RCA; shorter than 100 % are **HIGH**. Duration KPI is colour-coded in the comparative report. |
| `--expected-drain MAH_HR` | float | — | Expected battery drain rate in mAh/hr (e.g. `35.0`). Used as the anomaly detection baseline **instead of** the computed average. Any hour exceeding 1.5 × this value is highlighted as an anomaly. Also shown as a blue dashed reference line on graphs. |
| `--warn-drain MAH_HR` | float | `40` | Drain rate mAh/hr at or above which KPI cards and bar charts show **WARNING** (orange). |
| `--crit-drain MAH_HR` | float | `80` | Drain rate mAh/hr at or above which KPI cards and bar charts show **CRITICAL** (red). |
| `--expected-capacity MAH` | int | — | Override the battery capacity in mAh (e.g. `4500`). Useful when bugreport capacity parsing returns 0. |
| `--warn-temp C` | float | `45` | Thermal zone temperature °C at or above which RCA shows **WARNING**. |
| `--crit-temp C` | float | `80` | Thermal zone temperature °C at or above which RCA shows **CRITICAL**. |
| `--min-anr N` | int | `1` | ANR count at or above which RCA flags the issue. |
| `--min-crash N` | int | `1` | Crash count at or above which RCA flags the issue. |
| `--min-lmk N` | int | `5` | LMK kill count at or above which RCA flags memory pressure. |

### Example 6 — Device-specific SLA thresholds (Zebra TC501L PTT standby)

```bash
python bugreport_analyzer.py RC1\bugreport.txt ST5\bugreport.txt ^
       -l RC1 ST5 ^
       -o C:\Reports\Sprint42 ^
       --expected-hours 20 ^
       --expected-drain 35 ^
       --warn-drain 35 ^
       --crit-drain 60 ^
       --expected-capacity 4647 ^
       --min-lmk 10
```

What this does:
- Flags any session shorter than **17 h** (85 % × 20 h) as CRITICAL in the RCA
- Uses **35 mAh/hr** as the anomaly baseline — any hour draining more than 52.5 mAh is highlighted
- Shows a **blue dashed reference line** at 35 mAh/hr on all battery graphs
- Colours KPI cards **orange** above 35 mAh/hr and **red** above 60 mAh/hr
- Forces battery capacity to **4647 mAh** regardless of what the bugreport reports
- Only flags memory pressure in the RCA when **≥ 10 LMK kills** are recorded

## What Each Section Analyses

### 🔋 Battery Overview

| Metric | Description |
|---|---|
| **Total Drain %** | Start % → End % for the whole session |
| **Computed Drain (mAh)** | From `Estimated power use` in batterystats |
| **Avg Drain Rate (mAh/hr)** | Total drain ÷ session duration |
| **Anomaly Hours** | Hours where drain exceeded 1.5× the anomaly baseline |

**Colour coding (defaults — override with `--warn-drain` / `--crit-drain`):**
- 🟢 `< 40 mAh/hr` — Normal standby drain
- 🟡 `40–80 mAh/hr` — Elevated (investigate wakelocks)
- 🔴 `> 80 mAh/hr` — Excessive (likely wakelock / CPU regression)

> **Tip:** For device-specific standby SLAs, pass `--warn-drain` and `--crit-drain` so all KPI
> cards, bar charts, and RCA verdicts reflect your actual pass/fail criteria rather than generic defaults.

---

### ⚡ Per-UID Power Breakdown

Shows the top 30 UIDs/apps by power consumption from Android's batterystats, broken down into:

- **CPU** — CPU time while app was in foreground/background
- **WiFi** — WiFi traffic scan costs
- **Wakelock** — Energy cost of preventing deep sleep
- **Sensors** — GPS, accelerometer, etc.

---

### 🔒 Kernel & App Wakelocks

| Column | Meaning |
|---|---|
| **Name** | Wakelock identifier (kernel driver name or Java tag) |
| **Duration** | Total time held during session |
| **Count** | Number of acquire/release cycles |

> **Red = wakelock held > 50% of session duration** — Primary suspect for drain.  
> **Orange = > 1 hour** — Worth investigating.

---

### 🚫 ANR Events (Application Not Responding)

The analyser scans for ANRs from three sources:

| Source | Description |
|---|---|
| **ActivityManager log** | `ANR in <package>` log line with reason and CPU stats |
| **DropBox** | `Subject: ANR in` entries from ANR drop files |
| **logcat** | All occurrences of `ANR in` with timestamp |

For each ANR, the report shows:
- Timestamp
- Process / package name
- Reason (e.g., "Input dispatching timed out", "Broadcast of Intent not dispatched")
- First 30 lines of the associated trace (if available)

---

### 💥 Crashes

Detected crash types:

| Type | Detection Method |
|---|---|
| **Java FATAL** | `FATAL EXCEPTION` in logcat |
| **Java Exception** | `java.lang.RuntimeException`, `java.lang.Error`, etc. |
| **Process Died** | `Process <pkg> (pid N) has died` |
| **Native Crash** | `SIGSEGV`, `SIGABRT`, `Fatal signal` |
| **DropBox** | `Subject: crash` or `native_crash` entries |

---

### 🗑 Low Memory Killer (LMK)

The LMK section shows processes killed by Android's memory management:

| Column | Meaning |
|---|---|
| **Process** | App/process killed |
| **RSS** | Resident memory at time of kill (kB) |
| **ADJ** | OOM adjustment score (higher = more likely to be killed) |
| **Source** | `kernel LMK` or `ActivityManager` |

> Many LMK kills indicate **memory pressure** — may cause service restarts affecting PTT/VoIP behaviour.

---

### 🌡 Thermal

Shows peak temperatures per thermal zone during the session:

| Status | Default Threshold | Override Flag |
|---|---|---|
| 🟢 OK | `< 45 °C` | `--warn-temp` |
| 🟡 WARM | `45–80 °C` | `--warn-temp` / `--crit-temp` |
| 🔴 HOT | `> 80 °C` — throttling likely | `--crit-temp` |

Also lists thermal throttle events from `ThermalService` and kernel messages.

---

### 🔍 Root Cause Summary

Auto-generated analysis at the bottom of each report. All thresholds used below are configurable via CLI flags.

| RCA Trigger | Severity | Controlling Flag |
|---|---|---|
| Session ended < 85 % of expected duration | 🔴 CRITICAL | `--expected-hours` |
| Session ended < 100 % of expected duration | 🟡 HIGH | `--expected-hours` |
| Drain exceeds `--expected-drain` | 🔴 CRITICAL / 🟡 HIGH | `--expected-drain` + `--crit-drain` |
| Drain exceeds `--crit-drain` (no expected set) | 🔴 CRITICAL | `--crit-drain` |
| Drain exceeds `--warn-drain` (no expected set) | 🟡 HIGH | `--warn-drain` |
| App wakelock held > 80 % of session | 🔴 CRITICAL | — |
| Kernel wakelock held > 50 % of session | 🟡 HIGH | — |
| ANR count ≥ threshold | 🟡 HIGH | `--min-anr` |
| Crash count ≥ threshold (fatal) | 🔴 CRITICAL | `--min-crash` |
| Crash count ≥ threshold (non-fatal) | 🟡 HIGH | `--min-crash` |
| LMK kill count ≥ threshold | 🟡 HIGH | `--min-lmk` |
| Thermal throttle events > 5 | 🟡 HIGH | — |
| UID CPU drain > 30 mAh | 🟡 HIGH | — |

## Understanding the Battery Graph

```
┌─────────────────────────────────────────────────────────────┐
│  TOP PANEL                                                   │
│  • Blue line  = battery % over session time                 │
│  • Bars = per-hour drain in mAh                             │
│      Red    = > --crit-drain threshold (default 80 mAh/hr)  │
│      Orange = > --warn-drain threshold (default 40 mAh/hr)  │
│      Green  = below warning threshold (normal)              │
│  • Blue dashed line = --expected-drain target (if set)      │
│  • Orange dotted line = session average drain               │
│  • Red shaded regions = anomaly hours                       │
│      (hours > 1.5× anomaly baseline;                        │
│       baseline = --expected-drain if set, else avg)         │
│  • Yellow dashed lines = screen on/off events               │
├─────────────────────────────────────────────────────────────┤
│  BOTTOM PANEL                                               │
│  • Stacked bars for top-15 UIDs/apps                        │
│      Red    = CPU usage                                     │
│      Blue   = WiFi                                          │
│      Orange = Wakelock                                      │
│      Purple = Other (sensors, audio, bluetooth)             │
└─────────────────────────────────────────────────────────────┘
```

The **comparison graph** shows each build in a separate panel side-by-side, with per-hour drain values annotated directly on the level curve.

The **comparative HTML report** additionally shows:
- **Expected vs Actual drain** banner listing each build's actual rate against `--expected-drain` with the absolute overage in mAh/hr and percentage
- **Expected session duration** banner showing whether each build met the `--expected-hours` SLA

## Troubleshooting

### ❌ `ModuleNotFoundError: No module named 'matplotlib'`

```bash
pip install matplotlib
# or skip graphs:
python bugreport_analyzer.py bugreport.txt --no-graph
```

---

### ❌ `File not found: path/to/bugreport.txt`

Make sure you provide the path to the **extracted** `bugreport-*.txt` file, not the `.zip` archive.  
To extract: `adb bugreport` produces a `.zip` — unzip it first and locate `bugreport-<device>-<date>.txt`.

---

### ⚠ "No history entries found" / empty battery graph

The bugreport may not contain battery history. Check:
1. The bugreport was captured with `adb bugreport` (not just `dumpsys batterystats`)
2. Battery history is enabled on the device:  
   ```bash
   adb shell dumpsys batterystats --enable full-history
   ```
3. The bugreport `.txt` file contains `DUMP OF SERVICE batterystats` section

---

### ⚠ Battery drain shows 0% or unrealistic values

- Ensure the bugreport was captured **after a meaningful standby period** (not immediately after boot)
- Check if the device was on USB charging during the bugreport — charging sessions won't show drain
- The `battery_discharge:` event line must be present in the file

---

### ⚠ No ANRs/Crashes detected even though device had issues

- ANR and crash data lives in the `logcat` section and DropBox section of the bugreport
- If the bugreport was captured too long after the event, the logcat buffer may have rolled over
- Increase logcat buffer size before reproducing:  
  ```bash
  adb logcat -G 64M
  ```

---

### 🐢 Script runs slowly on large bugreports

Large bugreports (>200 MB) load into memory for single-pass parsing.  
Options:
- Use `--no-graph` to skip matplotlib (saves ~30% time)
- Ensure you have at least 2 GB of free RAM
