# Android BugReport Analyzer

> **Generic, multi-build, command-line tool for deep analysis of Android `bugreport.txt` files.**  
> Produces dark-themed HTML reports and PNG graphs for battery drain, ANRs, crashes, LMK, thermal events, and wakelocks.

## CLI Reference

```
python bugreport_analyzer.py <path1> [path2 ...] [OPTIONS]
```

### Positional Arguments

| Argument | Description |
|---|---|
| `path1 [path2 ...]` | One or more paths to `bugreport*.txt` files (full or relative paths) |

### Options

| Flag | Short | Default | Description |
|---|---|---|---|
| `--labels L1 L2 вҖҰ` | `-l` | `Build1, Build2 вҖҰ` | Short labels for each bugreport (must match count of paths) |
| `--output DIR` | `-o` | Directory of first bugreport | Output directory for all generated files |
| `--title "TEXT"` | `-t` | Auto-generated from labels + date | Title shown in the HTML report header |
| `--no-graph` | вҖ” | (graphs generated by default) | Skip PNG generation вҖ” faster when matplotlib is unavailable |
| `--help` | `-h` | вҖ” | Show help message |

## What Each Section Analyses

### рҹ”Ӣ Battery Overview

| Metric | Description |
|---|---|
| **Total Drain %** | Start % вҶ’ End % for the whole session |
| **Computed Drain (mAh)** | From `Estimated power use` in batterystats |
| **Avg Drain Rate (mAh/hr)** | Total drain Г· session duration |
| **Anomaly Hours** | Hours where drain exceeded 1.5Г— average |

**Colour coding:**
- рҹҹў `< 40 mAh/hr` вҖ” Normal standby drain
- рҹҹЎ `40вҖ“80 mAh/hr` вҖ” Elevated (investigate wakelocks)
- рҹ”ҙ `> 80 mAh/hr` вҖ” Excessive (likely wakelock / CPU regression)

---

### вҡЎ Per-UID Power Breakdown

Shows the top 30 UIDs/apps by power consumption from Android's batterystats, broken down into:

- **CPU** вҖ” CPU time while app was in foreground/background
- **WiFi** вҖ” WiFi traffic scan costs
- **Wakelock** вҖ” Energy cost of preventing deep sleep
- **Sensors** вҖ” GPS, accelerometer, etc.

---

### рҹ”’ Kernel & App Wakelocks

| Column | Meaning |
|---|---|
| **Name** | Wakelock identifier (kernel driver name or Java tag) |
| **Duration** | Total time held during session |
| **Count** | Number of acquire/release cycles |

> **Red = wakelock held > 50% of session duration** вҖ” Primary suspect for drain.  
> **Orange = > 1 hour** вҖ” Worth investigating.

---

### рҹҡ« ANR Events (Application Not Responding)

The analyser scans for ANRs from three sources:

| Source | Description |
|---|---|
| **ActivityManager log** | `ANR in <package>` log line with reason and CPU stats |
| **DropBox** | `Subject: ANR in` entries from ANR drop files |
| **logcat** | All occurrences of `ANR in` with timestamp |

For each ANR, the report shows:
- Timestamp
- Process / package name
- Reason (e.g., "Input dispatching timed out", "Broadcast of Intent not dispatched")
- First 30 lines of the associated trace (if available)

---

### рҹ’Ҙ Crashes

Detected crash types:

| Type | Detection Method |
|---|---|
| **Java FATAL** | `FATAL EXCEPTION` in logcat |
| **Java Exception** | `java.lang.RuntimeException`, `java.lang.Error`, etc. |
| **Process Died** | `Process <pkg> (pid N) has died` |
| **Native Crash** | `SIGSEGV`, `SIGABRT`, `Fatal signal` |
| **DropBox** | `Subject: crash` or `native_crash` entries |

---

### рҹ—‘ Low Memory Killer (LMK)

The LMK section shows processes killed by Android's memory management:

| Column | Meaning |
|---|---|
| **Process** | App/process killed |
| **RSS** | Resident memory at time of kill (kB) |
| **ADJ** | OOM adjustment score (higher = more likely to be killed) |
| **Source** | `kernel LMK` or `ActivityManager` |

> Many LMK kills indicate **memory pressure** вҖ” may cause service restarts affecting PTT/VoIP behaviour.

---

### рҹҢЎ Thermal

Shows peak temperatures per thermal zone during the session:

| Status | Threshold |
|---|---|
| рҹҹў OK | `< 45 В°C` |
| рҹҹЎ WARM | `45вҖ“80 В°C` |
| рҹ”ҙ HOT | `> 80 В°C` вҖ” throttling likely |

Also lists thermal throttle events from `ThermalService` and kernel messages.

---

### рҹ”Қ Root Cause Summary

Auto-generated analysis at the bottom of each report. Highlights:

- Wakelocks held > 80% of session вҶ’ prevents deep sleep вҶ’ battery drain
- High-drain UIDs exceeding expected standby rates
- ANR / crash counts
- LMK memory pressure
- Thermal throttling

## Troubleshooting

### вқҢ `ModuleNotFoundError: No module named 'matplotlib'`

```bash
pip install matplotlib
# or skip graphs:
python bugreport_analyzer.py bugreport.txt --no-graph
```

---

### вқҢ `File not found: path/to/bugreport.txt`

Make sure you provide the path to the **extracted** `bugreport-*.txt` file, not the `.zip` archive.  
To extract: `adb bugreport` produces a `.zip` вҖ” unzip it first and locate `bugreport-<device>-<date>.txt`.

---

### вҡ  "No history entries found" / empty battery graph

The bugreport may not contain battery history. Check:
1. The bugreport was captured with `adb bugreport` (not just `dumpsys batterystats`)
2. Battery history is enabled on the device:  
   ```bash
   adb shell dumpsys batterystats --enable full-history
   ```
3. The bugreport `.txt` file contains `DUMP OF SERVICE batterystats` section

---

### вҡ  Battery drain shows 0% or unrealistic values

- Ensure the bugreport was captured **after a meaningful standby period** (not immediately after boot)
- Check if the device was on USB charging during the bugreport вҖ” charging sessions won't show drain
- The `battery_discharge:` event line must be present in the file

---

### вҡ  No ANRs/Crashes detected even though device had issues

- ANR and crash data lives in the `logcat` section and DropBox section of the bugreport
- If the bugreport was captured too long after the event, the logcat buffer may have rolled over
- Increase logcat buffer size before reproducing:  
  ```bash
  adb logcat -G 64M
  ```

---

### рҹҗў Script runs slowly on large bugreports

Large bugreports (>200 MB) load into memory for single-pass parsing.  
Options:
- Use `--no-graph` to skip matplotlib (saves ~30% time)
- Ensure you have at least 2 GB of free RAM
